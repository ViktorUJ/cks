How Spot.io (Ocean) Was Used

  • Replaced Cluster Autoscaler + ASG (slow, rigid)
  • Automatic instance type selection
  • Spot/On-Demand switching
  • Virtual Node Groups (VNG) — logical node groups
  • "% of savings" pricing — transparent, aligned incentives

It worked well… until infrastructure grew

Problem 1: Job Node Consolidation

Cron job nodes were consolidated while tasks were running:

Task starts → Node provisioned → More tasks → More nodes
↓
Spot.io consolidates → Tasks interrupted → Restart
↓
New nodes → Consolidation again → ♻️ Loop

restrict-scale-down label existed but didn't prevent bin-packing consolidation

Problem 2: On-Demand Fallback & Oversizing

Stuck On-Demand

  • Nodes stayed On-Demand for weeks
  • Even when Spot capacity returned
  • No transparent tracking

Oversized Instances

  • Pod needs 1 vCPU
  • Ocean provisions 64 vCPU Spot
  • ~1-2% utilization for days
  • Spot.io charges % on "savings" of 64 vCPU

A small On-Demand would have been cheaper than Spot 64 vCPU + commission

1-2 week observation period at each phase

Spot.io + Karpenter in Parallel

  • Both controllers in the same cluster
  • Label/taint separation — no conflicts
  • Migrated workloads → nodeSelector for Karpenter nodes
  • Remaining workloads → VNG labels (Spot.io)
  • Rollback at any moment — just switch nodeSelector

Only 1 workload needed temporary rollback (GPU instance edge case)

Disruption Budgets — NodePool Config

apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: production
spec:
  disruption:
    consolidationPolicy: WhenEmptyOrUnderutilized
    consolidateAfter: 5m
    budgets:
      - nodes: 10%
      - schedule: "0 9 * * mon-fri"
        duration: 8h
        nodes: "0"    # No disruption during business hours

Disruption Budgets — Key Points

  • PDB on Deployment level — first line of defense
  • NodePool budgets — rate-limit node replacement
  • Schedule-based windows — protect peak hours
  • nodes: 10% — max 10% replaced simultaneously
  • nodes: "0" — complete ban during business hours
  • consolidateAfter: 5m — wait before consolidation

Managing Disruptions — Key Tools

  • 🛡️ PodDisruptionBudget (PDB) on Deployment — first line of defense, Karpenter respects this
  • ⚙️ NodePool budgets — control how many nodes can be replaced at the same time
  • 📅 Schedules — protect busy times from node replacements
  • 🔗 Affinity and Anti-Affinity — spread critical pods across different nodes, so they are not all on the same node

Together, these tools let you control when and how nodes are replaced safely and keep critical services running

10 Tips for Migration

  • Start with PoC on dev cluster
  • Document all VNGs first
  • Use NodePool cascade with weights
  • Set up monitoring before migration
  • Migrate in phases (Dev → Prod)
  • Run Spot.io + Karpenter in parallel
  • Min 10-15 instance types per NodePool
  • Don't forget PDB
  • Test Spot interruption with AWS FIS
  • Plan months, not weekends

Key Takeaways

  • 💰 Cost reduction — no license + better right-sizing
  • 🔍 Full transparency — logs, metrics, no black box
  • 🔧 More flexibility — NodePool, Disruption Budgets, weights
  • 🛡️ More control — config as K8s manifests, no SaaS dependency
  • 📉 Fewer incidents — no stuck OD, no consolidation loops

Karpenter isn't perfect — but for teams willing to invest in config & monitoring, it's a powerful tool.

k8i — Console Node Visualization

  • Replacement for Spot.io node dashboard
  • Real-time data directly from Kubernetes API
  • Per-node: pods, CPU, memory, utilization, Spot/OD, instance type
  • Filter by labels: k8i --labels karpenter.sh/nodepool=c8g
  • Sort by any column, group by pool
  • No external backend, no caching lag — actual cluster state
