# Лаба 112 — эталонное решение (etcd: бэкап и восстановление)

Все действия с etcd — на control plane ноде (`ssh k8s1_controlPlane_1`, `sudo -i`).
`etcdctl` уже установлен; сертификаты — в `/etc/kubernetes/pki/etcd/`.

## Задание 1. Снять снапшот etcd и скопировать на worker

```bash
# на control plane
ETCDCTL_API=3 etcdctl snapshot save /root/etcd-backup.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

ETCDCTL_API=3 etcdctl snapshot status /root/etcd-backup.db --write-out=table
```

Скопировать снапшот на рабочую машину (для проверки), например через `scp`:

```bash
# с worker PC
scp k8s1_controlPlane_1:/root/etcd-backup.db /var/work/tests/artifacts/etcd/etcd-backup.db
```

## Задание 2. Восстановить etcd из «старого» снапшота

На control plane лежит заранее снятый снапшот `/root/etcd-backup_old.db`.

```bash
# на control plane
ETCDCTL_API=3 etcdctl snapshot restore /root/etcd-backup_old.db \
  --data-dir=/var/lib/etcd-restore

# остановить etcd (убрать манифест статик-пода)
mv /etc/kubernetes/manifests/etcd.yaml /tmp/etcd.yaml

# в манифесте поменять hostPath каталога данных на /var/lib/etcd-restore
vim /tmp/etcd.yaml     # volumes: ... hostPath.path: /var/lib/etcd-restore

# вернуть манифест — kubelet поднимет etcd на восстановленных данных
mv /tmp/etcd.yaml /etc/kubernetes/manifests/

# подождать, проверить здоровье
kubectl get --raw='/healthz'
kubectl get pods -n kube-system
```

Кластер должен вернуться в рабочее состояние: API отвечает `ok`, поды `kube-system`
в статусе Running.
