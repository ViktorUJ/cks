# Лаба 124 — эталонное решение (HA control plane из 3 нод)

Дан кластер с одним control plane `cp1`, инициализированным через
`--control-plane-endpoint` и `--upload-certs`, с установленным CNI. Нужно присоединить
**две** control-plane ноды `cp2` и `cp3` (prerequisites и пакеты на них уже стоят), чтобы
получить нечётный кворум etcd (3 узла).

Официальная документация:
[Creating Highly Available clusters with kubeadm](https://kubernetes.io/docs/setup/production-environment/tools/kubeadm/high-availability/),
[kubeadm join](https://kubernetes.io/docs/reference/setup-tools/kubeadm/kubeadm-join/).

## Шаг 1. Получить certificate-key и join-команду (на `cp1`)

```bash
# 1. Загрузить сертификаты control plane в secret и получить certificate-key
sudo kubeadm init phase upload-certs --upload-certs
# ... печатает: <certificate-key>

# 2. Получить базовую команду join (токен + discovery-hash)
sudo kubeadm token create --print-join-command
# kubeadm join <endpoint>:6443 --token ... --discovery-token-ca-cert-hash sha256:...
```

Соберите команду для control-plane ноды, добавив `--control-plane` и `--certificate-key`.

## Шаг 2. Присоединить cp2 (задание 1)

```bash
ssh cp2
sudo kubeadm join <endpoint>:6443 \
  --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash> \
  --control-plane \
  --certificate-key <certificate-key>
exit
```

## Шаг 3. Присоединить cp3 (задание 2)

Тем же способом. Если с момента шага 1 прошло больше ~2 часов, `certificate-key` истёк —
повторите `upload-certs`; если истёк токен (~24 ч) — `kubeadm token create` заново.

```bash
ssh cp3
sudo kubeadm join <endpoint>:6443 \
  --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash> \
  --control-plane \
  --certificate-key <certificate-key>
exit
```

Проверка на `cp1`:

```bash
kubectl get nodes -l node-role.kubernetes.io/control-plane   # 3 ноды
kubectl get nodes                                            # все Ready
```

## Шаг 4. Проверить кворум etcd (задание 3)

При stacked-топологии на каждой control-plane ноде поднимается свой static pod `etcd-*`:

```bash
kubectl -n kube-system get pods -l component=etcd -o wide
# etcd-cp1   Running
# etcd-cp2   Running
# etcd-cp3   Running

# число членов etcd напрямую (с сертификатами, глава 37):
sudo ETCDCTL_API=3 etcdctl member list \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key
# 3 строки — три члена кластера etcd
```

## Почему именно 3

3 члена etcd имеют кворум 2 и переживают отказ **одного** узла. 2 члена переживают
**ноль** отказов (кворум 2 из 2) — это хуже одного узла. Поэтому HA строят на нечётном
числе: 3 (стандарт) или 5 (переживает отказ двух). Чётное число невыгодно (глава 35A).
