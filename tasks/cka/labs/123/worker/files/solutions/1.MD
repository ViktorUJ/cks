# Лаба 123 — эталонное решение (установка CNI с нуля + низкоуровневая сеть)

Кластер поднят **без CNI**: `kubectl get nodes` → все ноды `NotReady`, поды в
`netlab` висят в `Pending`/`ContainerCreating`, CoreDNS не поднимается. Пока нет
сетевого плагина — нет сети подов.

Подключение к нодам: `ssh k8s1_controlPlane_1`, `ssh k8s1_node_node_1`.

## Задание 1. Установить CNI руками

Официальный шаг kubeadm — «Installing a Pod network add-on»
([документация kubeadm](https://kubernetes.io/docs/setup/production-environment/tools/kubeadm/create-cluster-kubeadm/#pod-network),
[список аддонов](https://kubernetes.io/docs/concepts/cluster-administration/addons/)).
Ставим Calico ([Calico quickstart](https://docs.tigera.io/calico/latest/getting-started/kubernetes/quickstart)):

```bash
# версию манифеста берите совместимую с k8s 1.35 (см. docs.tigera.io)
kubectl apply -f https://raw.githubusercontent.com/projectcalico/calico/v3.28.2/manifests/calico.yaml

# наблюдаем, как поднимается сеть
kubectl -n kube-system get pods -l k8s-app=calico-node -w
kubectl get nodes -w        # ноды переходят в Ready
```

После установки CNI:
- ноды становятся `Ready`;
- CoreDNS поднимается (`kubectl -n kube-system get deploy coredns`);
- поды `netprobe` в `netlab` разъезжаются по двум нодам и переходят в `Running`.

> Если хотите, чтобы пул адресов Calico совпадал с `--pod-network-cidr=10.0.0.0/16`,
> задайте `CALICO_IPV4POOL_CIDR=10.0.0.0/16` в манифесте перед `apply` (на прохождение
> лабы это не влияет).

## Задание 2-3 (проверяются автоматически)

Как только CNI заработал, CoreDNS и `netprobe` (2 пода на разных нодах) поднимаются
сами — это и есть подтверждение работающей сети подов, в т.ч. **меж-нодовой**.

## Задание 4. Отчёт по низкоуровневой сети → `net-lowlevel.txt`

Собираем факты на control plane ноде:

```bash
ssh k8s1_controlPlane_1

# конфиг CNI, который положил плагин
sudo ls /etc/cni/net.d
# напр.: 10-calico.conflist

# устройство маршрута по умолчанию
ip route show default
# default via 10.10.1.1 dev ens5 ...   -> dev = ens5
```

На рабочей машине:

```bash
mkdir -p /home/ubuntu/answers
cat > /home/ubuntu/answers/net-lowlevel.txt <<EOF
cni_conf=10-calico.conflist
default_route_dev=ens5
EOF
```

> Подставьте свои значения из выводов выше.

## Разбор network namespaces и маршрутов (низкоуровнево)

Полезно понять, как CNI связывает под с сетью ноды. На ноде, где есть под:

```bash
# найти контейнеры и их сетевые namespace
sudo crictl ps
sudo crictl inspectp <pod-id> | grep -i netns    # путь к сетевому namespace пода

# список сетевых namespace
ip netns list
# зайти в netns процесса пода и посмотреть его сеть
PID=$(sudo crictl inspect <container-id> | jq '.info.pid')
sudo nsenter -t $PID -n ip addr        # интерфейс eth0 пода и его IP
sudo nsenter -t $PID -n ip route       # маршрут по умолчанию пода

# veth-пары на хосте (один конец в поде, другой на ноде)
ip -o link | grep -E 'cali|veth'
# маршруты к подовым сетям (Calico добавляет маршруты на другие ноды)
ip route | grep -E '10\.|192\.168'
```

Идея: CNI создаёт для пода отдельный **network namespace**, соединяет его veth-парой с
хостом, выдаёт IP из пула и прописывает **маршруты** к сетям подов на других нодах —
так работает «плоская сеть» Kubernetes.

Документация: [Cluster Networking](https://kubernetes.io/docs/concepts/cluster-administration/networking/),
[Network Plugins (CNI)](https://kubernetes.io/docs/concepts/extend-kubernetes/compute-storage-net/network-plugins/).
