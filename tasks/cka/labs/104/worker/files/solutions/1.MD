## Задание 1. Под `alpine` на ноду с меткой `disk=ssd`

```yaml
apiVersion: v1
kind: Pod
metadata: {name: alpine}
spec:
  nodeSelector: {disk: ssd}
  containers:
  - name: alpine
    image: alpine:3.15
    command: ["sleep", "6000"]
```

```bash
kubectl apply -f alpine.yaml
kubectl get po alpine -o wide
```

## Задание 2. Taint ноды + под `alpha` с toleration

```bash
# ставим taint на worker-ноду (с меткой disk=ssd)
NODE=$(kubectl get nodes -l disk=ssd -o jsonpath='{.items[0].metadata.name}')
kubectl taint nodes $NODE app_type=alpha:NoSchedule
```

```yaml
apiVersion: v1
kind: Pod
metadata: {name: alpha}
spec:
  tolerations:
  - key: app_type
    operator: Equal
    value: alpha
    effect: NoSchedule
  containers:
  - {name: alpha, image: redis}
```

```bash
kubectl apply -f alpha.yaml
```

## Задание 3. Static pod `static-busybox` на control-plane

На control-plane ноде (по SSH) положить манифест в `/etc/kubernetes/manifests/`:

```yaml
# /etc/kubernetes/manifests/static-busybox.yaml
apiVersion: v1
kind: Pod
metadata:
  name: static-busybox
  labels: {pod-type: static-pod}
spec:
  containers:
  - name: static-busybox
    image: busybox
    command: ["sleep", "60000"]
```

kubelet сам поднимет под; в API он появится как `static-busybox-<имя-ноды>`.

## Задание 6. Deployment с PodAntiAffinity + PDB

```yaml
apiVersion: apps/v1
kind: Deployment
metadata: {name: important-app2, namespace: app2-system}
spec:
  replicas: 3
  selector: {matchLabels: {app: important-app2}}
  template:
    metadata: {labels: {app: important-app2}}
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector: {matchLabels: {app: important-app2}}
            topologyKey: kubernetes.io/hostname
      tolerations:
      - key: node-role.kubernetes.io/control-plane
        operator: Exists
        effect: NoSchedule
      containers:
      - {name: important-app2, image: nginx}
---
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata: {name: important-app2, namespace: app2-system}
spec:
  minAvailable: 1
  selector: {matchLabels: {app: important-app2}}
```

```bash
kubectl create namespace app2-system
kubectl apply -f app2.yaml
```

## Задание 7. Строгое распределение через topologySpreadConstraints

```yaml
apiVersion: apps/v1
kind: Deployment
metadata: {name: spread-app}
spec:
  replicas: 2
  selector: {matchLabels: {app: spread-app}}
  template:
    metadata: {labels: {app: spread-app}}
    spec:
      topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: kubernetes.io/hostname
        whenUnsatisfiable: DoNotSchedule       # СТРОГО
        labelSelector:
          matchLabels: {app: spread-app}
      tolerations:
      - key: node-role.kubernetes.io/control-plane
        operator: Exists
        effect: NoSchedule
      containers:
      - {name: spread-app, image: nginx}
```

```bash
kubectl apply -f spread.yaml
kubectl get po -l app=spread-app -o wide     # по одному на ноду
```

**Строгий vs мягкий.** `whenUnsatisfiable: DoNotSchedule` — строго: если ровно разложить
нельзя (нод меньше, чем нужно), лишний под останется `Pending`. Чтобы поды всё же
запустились с допуском на перекос, замените на **мягкий** режим:

```yaml
        whenUnsatisfiable: ScheduleAnyway      # МЯГКО (с допуском)
```

Аналогично у podAntiAffinity: `requiredDuringScheduling...` — строго (риск Pending),
`preferredDuringScheduling...` (с `weight`) — мягко (все поды запустятся).
