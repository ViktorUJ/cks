# Глава 8. Устойчивость: fault injection, timeouts, retries, circuit breaking

> **Что дальше.** Сеть ненадёжна: сервисы тормозят, перезагружаются, отдают ошибки. В
> этой главе разберём, как Istio делает приложение устойчивым к таким сбоям - и всё
> на уровне инфраструктуры, без изменения кода. Сначала научимся намеренно ломать
> сервис (fault injection), чтобы проверить устойчивость, а потом чинить: таймауты,
> ретраи и circuit breaking.

## 8.1. Проблема: сбои и каскадные отказы

Когда один сервис обращается к другому по сети, всё может пойти не так: получатель
тормозит, отдаёт 503 или вообще недоступен. Если это не обрабатывать, беда расползается: медленный
сервис задерживает вызывающего, у того копятся соединения, и в итоге падает вся
цепочка. Это называют **каскадным отказом** (cascading failure).

Istio даёт набор инструментов против этого, и все они настраиваются в уже знакомых
ресурсах:

| Инструмент | Где настраивается | Что делает |
|------------|-------------------|------------|
| Fault injection | VirtualService | намеренно вносит задержки и ошибки для теста |
| Timeout | VirtualService | обрывает слишком долгий запрос |
| Retry | VirtualService | повторяет неудачный запрос |
| Circuit breaking | DestinationRule | ограничивает нагрузку и отсекает больные реплики |

## 8.2. Fault injection: ломаем намеренно

Прежде чем защищаться от сбоев, надо уметь их воспроизводить. Fault injection - это
контролируемое внесение ошибок, чтобы проверить, как система себя поведёт. Есть два
вида.

Fault injection задаётся в **`VirtualService`** для того сервиса, который мы хотим
«ломать» (в примерах ниже - `ping-pong`): в поле `hosts` указываем этот сервис, а в
`http.fault` - какой сбой вносить.

**Задержка (delay)** - имитируем медленный сервис:

```yaml
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: ping-pong
spec:
  hosts:
  - ping-pong               # к какому сервису применяем
  http:
  - fault:
      delay:
        fixedDelay: 5s
        percentage:
          value: 100        # добавить 5с задержки ко всем запросам
    route:
    - destination:
        host: ping-pong
```

**Обрыв (abort)** - имитируем ошибку:

```yaml
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: ping-pong
spec:
  hosts:
  - ping-pong
  http:
  - fault:
      abort:
        httpStatus: 503
        percentage:
          value: 50         # половине запросов сразу вернуть 503
    route:
    - destination:
        host: ping-pong
```

```mermaid
flowchart LR
    E["Envoy с fault"] -->|"половина: сразу 503"| X["до сервиса не дошло"]
    E -->|"половина: ок"| S["сервис ping-pong"]
    style E fill:#326ce5,color:#fff
    style X fill:#db4437,color:#fff
    style S fill:#0f9d58,color:#fff
```

Важный момент: при `abort` ошибку генерирует **сам Envoy**, запрос до реального сервиса
даже не доходит. Это удобно и безопасно: вы тестируете устойчивость вызывающей стороны,
не трогая код и не ломая сам сервис по-настоящему.

## 8.3. Timeout: обрываем долгий запрос

Если сервис отвечает слишком долго, лучше оборвать запрос, чем ждать бесконечно и
держать занятым соединение. Таймаут задаётся в `VirtualService` для нужного сервиса
(ниже показан только блок `http`, полная структура - как в примере из 8.2):

```yaml
http:
- timeout: 3s           # ждём ответа не больше 3 секунд
  route:
  - destination:
      host: reviews
```

Если `reviews` не ответил за 3 секунды, Envoy обрывает запрос и возвращает вызывающему
ошибку (`504`). Без таймаута один медленный сервис может «подвесить» всю цепочку.

## 8.4. Retry: повторяем неудачный запрос

Многие сбои временные: под перезапустился, была секундная сетевая проблема. В таких
случаях простой повтор запроса решает проблему. Ретраи тоже задаются в `VirtualService`
(ниже - только блок `http`):

```yaml
http:
- retries:
    attempts: 3               # до 3 повторных попыток
    perTryTimeout: 2s         # таймаут на каждую попытку
    retryOn: 5xx,connect-failure   # при каких ошибках повторять
  route:
  - destination:
      host: reviews
```

```mermaid
flowchart LR
    E["Envoy вызывателя"] -->|"попытка 1: 503"| S["reviews"]
    E -->|"повтор: 200"| S
    style E fill:#326ce5,color:#fff
    style S fill:#0f9d58,color:#fff
```

Разберём поля:

- **`attempts`** - сколько раз повторить после первой неудачи.
- **`perTryTimeout`** - таймаут на каждую отдельную попытку.
- **`retryOn`** - при каких условиях повторять: `5xx` (любой 5xx-ответ),
  `connect-failure`, `gateway-error`, `retriable-4xx` и другие, через запятую.

Ретраи заметно повышают надёжность. Простая математика: если сервис ошибается в 50%
случаев, то при 3 ретраях вероятность, что все 4 попытки провалятся, равна
0.5 в степени 4 = ~6%. То есть успех вырастает с 50% до ~94%, и всё это незаметно для
приложения.

### Подводные камни ретраев

Ретраи мощны, но у них есть тонкости, о которых важно помнить.

- **Istio по умолчанию уже ретраит.** Даже без блока `retries` Istio применяет к
  HTTP-запросам ретраи по умолчанию (обычно `attempts: 2` на «безопасные» сбои вроде
  `connect-failure`, `refused-stream`, `unavailable`). Явный `retries` переопределяет это.
  То есть «ретраев нет» - миф; вопрос лишь в том, свои у вас настройки или дефолтные.
- **Ретраить можно только идемпотентные операции.** Повтор `GET` безопасен. А повтор
  `POST`, который создаёт заказ или списывает деньги, при ретраях выполнится дважды.
  Ретраи для не-идемпотентных запросов включайте осознанно (или не включайте) - это та же
  проблема, что с зеркалированием из главы 6.
- **Осторожно с retry storm (лавина ретраев).** Если ошибается вся цепочка, каждый слой
  начинает ретраить - и нагрузка умножается, добивая уже перегруженный сервис. Держите
  `attempts` небольшим (2-3) и ограничивайте одновременные ретраи через
  `connectionPool.http.maxRetries` в DestinationRule.
- **Таймаут должен вмещать все попытки.** Общий `timeout` запроса отсчитывается на все
  ретраи сразу. Если `timeout: 3s`, а `perTryTimeout: 2s` при `attempts: 3`, то на вторую
  и третью попытку времени уже не останется. Согласуйте `timeout ≈ attempts ×
  perTryTimeout` (плюс запас).

## 8.5. Где ставить ретраи: важная тонкость

Ретраи настраивают на стороне сервиса, который **делает запрос** (клиента), а не на
стороне сервиса, который отвечает с ошибкой. Причина простая: повторяет запрос тот
Envoy, который делает исходящий вызов.

Вспомните пример из лабы 03: `frontend` обращается к `ping-pong`, а на `ping-pong` включён
fault injection (50% ошибок). Ретраи надо ставить в VirtualService для `frontend` -
тогда его Envoy будет повторять исходящие вызовы к `ping-pong`.

Ставить ретраи в VirtualService для `ping-pong` было бы бессмысленно: там сидит сам
fault injection, и Envoy повторял бы сгенерированную им же ошибку - бесконечный
бессмысленный цикл.

Проверить, что ретраи реально происходят, можно по метрикам Envoy вызывающего пода:

```bash
kubectl exec -it <frontend-pod> -c istio-proxy -- \
  pilot-agent request GET stats | grep upstream_rq_retry
```

## 8.6. Circuit breaking: пул соединений

Ретраи и таймауты работают с отдельным запросом. Circuit breaking (предохранитель)
работает на уровне сервиса: он ограничивает, сколько запросов и соединений разрешено
слать получателю. Настраивается в DestinationRule через `connectionPool`.

```yaml
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: reviews-dr
spec:
  host: reviews
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100          # максимум TCP-соединений
      http:
        http1MaxPendingRequests: 10  # максимум запросов в очереди
        maxRequestsPerConnection: 10
```

Смысл в том, чтобы не «завалить» перегруженный сервис. Когда лимиты превышены, Envoy
сразу отклоняет лишние запросы (`503`), не ставя их в бесконечную очередь. Это даёт
сервису шанс разгрестись, а вызывающему - быстро получить ответ (пусть и ошибку)
вместо зависания. Лучше быстро отказать, чем медленно умирать всей цепочкой.

Полезные поля `connectionPool`:

- `tcp.maxConnections` - потолок TCP-соединений к сервису;
- `http.http1MaxPendingRequests` - сколько запросов может ждать в очереди;
- `http.http2MaxRequests` - максимум одновременных запросов (актуально для HTTP/2 и gRPC,
  где всё идёт по одному соединению - глава 10);
- `http.maxRequestsPerConnection` - после скольких запросов переоткрывать соединение;
- `http.maxRetries` - потолок одновременных ретраев ко всему сервису (защита от retry
  storm);
- `tcp.connectTimeout` / `http.idleTimeout` - таймауты установки и простоя соединения.

## 8.7. Outlier detection: отсекаем больные реплики

Вторая часть circuit breaking - `outlierDetection`. Она следит за отдельными
репликами и временно исключает из балансировки те, что сыпят ошибками.

```yaml
  trafficPolicy:
    outlierDetection:
      consecutive5xxErrors: 5    # 5 ошибок 5xx подряд
      interval: 10s              # как часто проверять
      baseEjectionTime: 30s      # на сколько исключить реплику
      maxEjectionPercent: 50     # но не больше 50% реплик разом
```

```mermaid
flowchart TB
    LB["Балансировка reviews"] --> P1["реплика 1 (ок)"]
    LB --> P2["реплика 2 (ок)"]
    LB -.->|"исключена"| P3["реплика 3 (больная)"]
    style LB fill:#673ab7,color:#fff
    style P1 fill:#0f9d58,color:#fff
    style P2 fill:#0f9d58,color:#fff
    style P3 fill:#db4437,color:#fff
```

Логика: если реплика выдала `consecutive5xxErrors` ошибок подряд, Envoy на
`baseEjectionTime` убирает её из пула и шлёт трафик только на здоровые. Через это время
реплику возвращают и снова проверяют. `maxEjectionPercent` не даёт исключить слишком
много реплик сразу, чтобы не остаться без рабочих.

Отдельно вспомните главу 7: именно `outlierDetection` нужен для locality failover -
без него Istio не понимает, что реплики в зоне больны, и не переключает трафик.

### Как это сочетается с liveness/readiness probes

Outlier detection легко перепутать с probes Kubernetes, но это разные механизмы на
разных уровнях - и они дополняют друг друга.

| | Readiness / Liveness probes | Outlier detection |
|---|---|---|
| Кто проверяет | kubelet на ноде | Envoy вызывающего пода |
| Как | **активно** опрашивает health-эндпоинт пода | **пассивно** смотрит на реальные ответы (5xx, таймауты, обрывы) |
| На основании чего | что приложение само сообщает о себе | что реально вернулось на боевые запросы |
| Область действия | глобально: readiness убирает под из Endpoints - его не видит никто | локально: каждый вызывающий Envoy решает сам за себя |
| Скорость | период пробы + распространение Endpoints | сразу по факту ошибок |
| Действие | readiness - убрать из Endpoints; liveness - перезапустить контейнер | временно исключить эндпоинт из своего пула |

Как они работают вместе:

- **Readiness** - первый рубеж: если под сам объявил себя не готовым, kubelet убирает его
  из Endpoints сервиса, istiod перестаёт рассылать его как эндпоинт, и трафик к нему не
  идёт вообще - outlier detection его даже «не увидит».
- **Liveness** - если контейнер завис, kubelet перезапускает его; на время перезапуска под
  всё равно проваливает readiness и выпадает из Endpoints.
- **Outlier detection** закрывает то, что probes пропускают: под **проходит readiness**
  (говорит «я здоров»), но на деле сыпет ошибками - например, из-за упавшей зависимости
  или бага, который health-эндпоинт не ловит. Envoy видит реальные 5xx и временно
  исключает такую реплику из балансировки, не дожидаясь, пока приложение «признается».

Практический вывод: probes и outlier detection не заменяют, а **дополняют** друг друга.
Readiness/liveness - это «здоров ли я по собственной оценке», а outlier detection - «а как
я реально отвечаю на боевой трафик». Для отказоустойчивости (и для locality failover из
главы 7) нужны оба: корректные probes **плюс** `outlierDetection`.

> Нюанс Istio: у пода в mesh readiness-проба приложения объединяется с готовностью самого
> sidecar (`istio-proxy`, порт `15021`). Если sidecar не готов, под тоже не готов и
> выпадает из Endpoints (см. главу 4).

## 8.8. Best practices

- **Слоите защиту.** Таймаут + ретраи + circuit breaking работают вместе: таймаут не даёт
  висеть, ретраи прячут временные сбои, circuit breaking защищает перегруженный сервис. По
  отдельности каждый слабее.
- **Ставьте таймауты везде.** По умолчанию у Istio таймаута запроса нет - запрос может
  ждать сколько угодно. Задавайте разумный `timeout` на каждый вызов, иначе один медленный
  сервис подвесит всю цепочку.
- **Ретраьте только идемпотентное.** `GET` - да; `POST`/`PUT` с побочными эффектами - лишь
  если операция идемпотентна (или через ключ идемпотентности на стороне приложения).
- **Небольшой `attempts` + `maxRetries`.** 2-3 попыток достаточно; ограничивайте
  одновременные ретраи через `connectionPool.http.maxRetries`, чтобы не устроить retry
  storm.
- **Согласуйте таймаут и ретраи.** Общий `timeout` должен вмещать `attempts ×
  perTryTimeout`, иначе часть попыток не успеет выполниться.
- **Circuit breaking - консервативно и по нагрузке.** Лимиты `connectionPool` подбирайте
  под реальную ёмкость сервиса; лучше быстро отдать 503, чем копить очередь.
- **`outlierDetection` с `maxEjectionPercent`.** Исключайте больные реплики, но не все
  разом - иначе Envoy уйдёт в panic mode (глава 7) и снова начнёт слать трафик на всех.
- **Проверяйте устойчивость fault injection'ом.** Не верьте, что resilience-конфиг
  работает, пока не сломали сервис намеренно (`delay`/`abort`) и не увидели, что ретраи,
  таймауты и предохранители реально отрабатывают.

## 8.9. Итоги главы

- Ненадёжная сеть ведёт к каскадным отказам; Istio защищает от них на уровне
  инфраструктуры.
- **Fault injection** (`fault.delay`, `fault.abort`) в VirtualService намеренно вносит
  задержки и ошибки для проверки устойчивости; ошибку генерирует сам Envoy.
- **Timeout** в VirtualService обрывает слишком долгий запрос (возвращает 504).
- **Retry** в VirtualService повторяет неудачный запрос (`attempts`, `perTryTimeout`,
  `retryOn`); заметно повышает надёжность.
- Ретраи настраивают на стороне сервиса-клиента (который делает запрос), а не на стороне
  сервиса, который отвечает с ошибкой.
- Подводные камни ретраев: Istio ретраит по умолчанию (attempts 2), ретраить безопасно
  только идемпотентное, риск retry storm (ограничивать `attempts` и `maxRetries`), общий
  `timeout` должен вмещать все попытки.
- **Circuit breaking** в DestinationRule: `connectionPool` ограничивает нагрузку,
  `outlierDetection` исключает больные реплики.
- `outlierDetection` также необходим для locality failover (глава 7).
- Outlier detection (пассивная проверка Envoy по реальным ответам) и probes kubelet
  (активная проверка health-эндпоинта) дополняют друг друга: probes убирают под из
  Endpoints глобально, outlier detection ловит реплику, которая проходит readiness, но
  реально отвечает ошибками.
