# Lab 32 - Reference Solution

Istio `1.29.1` (default profile) is installed. Namespace `app` (istio-injection
enabled) runs the `viktoruj/ping_pong` image, which speaks gRPC
(`PingPong.Echo` returns the serving pod's hostname):

- **grpc-server** - the gRPC Echo/Health server, **3 replicas** (gRPC on port
  `8079`);
- **grpc-client** - the same image, used to drive gRPC load
  (`/app -grpc-client ...`).

The `grpc-server` **Service** is created with a **mis-named** port (`tcp`).
Because of that, Istio treats port `8079` as raw L4 TCP: the single long-lived
HTTP/2 gRPC connection is pinned to one pod, so the client sees only **one**
distinct server and per-request balancing does not happen.

## Goal

1. Make Istio recognise the gRPC port so it does **per-request** (HTTP/2) load
   balancing across all backend pods.
2. Add a `VirtualService` for `grpc-server` with gRPC **retries** and a
   **timeout**.

## Step 1. Fix the Service port naming

gRPC is HTTP/2, not raw TCP. Istio detects the protocol from the port **name
prefix** (`grpc`, `http2`, ...) or the explicit `appProtocol` field. Rename the
port to `grpc` (and/or set `appProtocol: grpc`):

```bash
kubectl -n app patch svc grpc-server --type=json -p='[
  {"op":"replace","path":"/spec/ports/0/name","value":"grpc"},
  {"op":"add","path":"/spec/ports/0/appProtocol","value":"grpc"}
]'
```

Equivalent full manifest:

```bash
kubectl apply -f - <<'EOF'
apiVersion: v1
kind: Service
metadata:
  name: grpc-server
  namespace: app
  labels:
    app: grpc-server
spec:
  selector:
    app: grpc-server
  ports:
    - name: grpc
      appProtocol: grpc
      port: 8079
      targetPort: 8079
EOF
```

As soon as Istio sees an HTTP/2 (gRPC) cluster, Envoy balances **each request**
inside the shared connection across all endpoints — no extra config required.

## Step 2. VirtualService with gRPC retries + timeout

gRPC rides on HTTP, so it is configured through the `http` block (not `tcp`):
retries, timeouts, header routing and per-request balancing all apply.

```bash
kubectl apply -f - <<'EOF'
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: grpc-server
  namespace: app
spec:
  hosts:
    - grpc-server
  http:
    - route:
        - destination:
            host: grpc-server
            port:
              number: 8079
      timeout: 2s
      retries:
        attempts: 3
        perTryTimeout: 1s
        retryOn: connect-failure,refused-stream,unavailable,cancelled,deadline-exceeded
EOF
```

- `retryOn` uses gRPC-aware conditions: `unavailable`, `cancelled`,
  `deadline-exceeded` map to gRPC status codes; `refused-stream` and
  `connect-failure` cover transport issues.
- `timeout` bounds the whole request; `perTryTimeout` bounds each attempt.

## Step 3. (optional) Pin the load-balancing policy

Per-request balancing is on by default once the port is gRPC. To make the
policy explicit:

```bash
kubectl apply -f - <<'EOF'
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: grpc-server
  namespace: app
spec:
  host: grpc-server
  trafficPolicy:
    loadBalancer:
      simple: ROUND_ROBIN
EOF
```

## Step 4. Verify

Drive gRPC load from the client and confirm requests hit **all three** backend
pods (per-request balancing):

```bash
kubectl exec -n app deploy/grpc-client -c ping-pong -- \
  /app -grpc-client -target grpc-server:8079 -n 180 -c 4
```

Expected tail:

```
--- summary ---
requests: 180  ok: 180  errors: 0
distinct servers: 3
host grpc-server-xxxx-aaaa: 60
host grpc-server-xxxx-bbbb: 60
host grpc-server-xxxx-cccc: 60
```

`distinct servers: 3` proves the requests are balanced per-request. Before the
fix (port named `tcp`) the same command reports `distinct servers: 1`.

## Why this works

- **gRPC is HTTP/2, not TCP.** With an L4/TCP view Envoy load-balances
  *connections*: one gRPC client keeps a single long-lived connection, so every
  call sticks to one pod. Declaring the port as `grpc` makes Envoy parse HTTP/2
  and balance **each request** (stream) across endpoints.
- **Port naming is the switch.** The port must be named `grpc` / `grpc-*` (or
  `http2`), or carry `appProtocol: grpc`. A neutral name (`tcp`, unnamed) drops
  every L7 feature: no per-request LB, no retries, no timeouts, no gRPC metrics.
- **L7 features apply to gRPC.** Because it is HTTP, gRPC gets `http` retries
  (with gRPC-aware `retryOn`), `timeout`/`perTryTimeout`, header routing, fault
  injection and rich telemetry — exactly like plain HTTP.
