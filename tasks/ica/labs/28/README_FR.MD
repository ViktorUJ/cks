# Lab 28 - TCP routing : routage du trafic non-HTTP

## Aperﾃｧu

Tout le trafic n'est pas du HTTP. Les bases de donnﾃｩes, les brokers, les protocoles
personnalisﾃｩs fonctionnent sur du TCP brut, oﾃｹ il n'y a ni host, ni path, ni en-tﾃｪtes.
Istio route ce trafic au niveau L4 : via un `Gateway` avec `protocol: TCP` et un
`VirtualService.tcp`, oﾃｹ la route est choisie selon le **port** du listener.

Dans ce lab est dﾃｩployﾃｩ un service TCP-echo `tcp-echo` en deux versions (sur le port TCP
brut `9000`) :
- **v1** rﾃｩpond avec le prﾃｩfixe `one` ;
- **v2** rﾃｩpond avec le prﾃｩfixe `two`.

L'ingress gateway ﾃｩcoute dﾃｩjﾃ le TCP sur le NodePort `31400`.

## Exercice

1. Crﾃｩer un `Gateway` avec un serveur `protocol: TCP` sur le port `31400`.
2. Crﾃｩer une `DestinationRule` avec les subsets `v1`/`v2`.
3. Crﾃｩer un `VirtualService` avec une route `tcp` (match sur le port 31400), rﾃｩpartissant
   de faﾃｧon pondﾃｩrﾃｩe les connexions entre v1 (80%) et v2 (20%).
4. Vﾃｩrifier que le TCP brut ﾃ travers le gateway atteint le service et que l'ﾃｩcho revient.

## ﾃ液ape 1. Gateway avec listener TCP

```bash
kubectl apply -f - <<'EOF'
apiVersion: networking.istio.io/v1
kind: Gateway
metadata:
  name: tcp-echo-gateway
  namespace: app
spec:
  selector:
    istio: ingressgateway
  servers:
    - port:
        number: 31400
        name: tcp
        protocol: TCP
      hosts:
        - "*"
EOF
```

## ﾃ液ape 2. DestinationRule avec subsets

```bash
kubectl apply -f - <<'EOF'
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: tcp-echo
  namespace: app
spec:
  host: tcp-echo
  subsets:
    - name: v1
      labels:
        version: v1
    - name: v2
      labels:
        version: v2
EOF
```

## ﾃ液ape 3. VirtualService avec route TCP

```bash
kubectl apply -f - <<'EOF'
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: tcp-echo
  namespace: app
spec:
  hosts:
    - "*"
  gateways:
    - tcp-echo-gateway
  tcp:
    - match:
        - port: 31400
      route:
        - destination:
            host: tcp-echo
            port:
              number: 9000
            subset: v1
          weight: 80
        - destination:
            host: tcp-echo
            port:
              number: 9000
            subset: v2
          weight: 20
EOF
```

## Comment ﾃｧa marche

- **Le routage TCP** fonctionne au niveau L4 : pas de host/path/en-tﾃｪtes HTTP, donc la
  route est choisie selon le **port du listener** (`match.port`). Un `Gateway` avec
  `protocol: TCP` ouvre un listener TCP ordinaire dans Envoy, et `VirtualService.tcp`
  dirige la connexion vers le bon subset.
- **Le nom du port est important** : le port du service/gateway doit s'appeler `tcp` (ou
  `tcp-*`). Istio dﾃｩtermine le protocole d'aprﾃｨs le prﾃｩfixe du nom de port ; un nom sans
  prﾃｩfixe ou `http-*` forcera Istio ﾃ le considﾃｩrer comme du HTTP et le protocole brut sera
  cassﾃｩ.
- **Le TCP pondﾃｩrﾃｩ** rﾃｩpartit les *connexions* (et non les requﾃｪtes) entre les subsets -
  chaque nouvelle connexion TCP est routﾃｩe selon le poids.
- Les fonctions L7 (retries, header routing, fault injection) **ne s'appliquent pas** aux
  routes TCP - seulement les politiques de niveau connexion (connection pool, timeouts)
  via la `DestinationRule`.

## Protocoles apparentﾃｩs

- **MongoDB/MySQL/Redis** - nommez le port `mongo-*` / `mysql-*` / `redis-*` pour qu'Envoy
  applique le bon parseur de protocole ; le routage passe malgrﾃｩ tout par des routes `tcp`.
- **WebSocket** - bien qu'il s'agisse d'une connexion longue durﾃｩe, il fonctionne
  par-dessus HTTP `Upgrade`, utilisez donc des routes `http` ordinaires et des noms de
  port `http-*`, et non du TCP.
